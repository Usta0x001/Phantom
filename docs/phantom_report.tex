\chapter*{Abstract}
\addcontentsline{toc}{chapter}{Abstract}

This report presents \phantom@{}, an autonomous AI-powered penetration testing platform that leverages large language models (LLMs) to conduct end-to-end security assessments with minimal human intervention. Unlike traditional vulnerability scanners that rely on static signatures and predefined rule sets, \phantom@{} employs a Reason-then-Act (ReAct) cognitive loop that enables it to analyze target environments, formulate attack strategies, chain multi-step exploits, and adapt its approach based on observed behavior.

The system integrates over 30 industry-standard security tools---including nmap, nuclei, sqlmap, ffuf, httpx, and katana---executing all operations within an ephemeral Docker sandbox that provides strict isolation from the host system. A multi-agent architecture enables parallel reconnaissance, exploitation, and verification tasks, while a 7-layer security framework ensures that the platform itself cannot be weaponized beyond its intended scope.

Key contributions of this work include:
\begin{itemize}[leftmargin=2em]
    \item An autonomous agent architecture achieving human-comparable penetration testing without manual intervention
    \item A sandboxed execution environment with defense-in-depth security controls
    \item Automated vulnerability verification producing working proof-of-concept exploits
    \item MITRE ATT\&CK enrichment and multi-framework compliance mapping (OWASP, PCI DSS, NIST)
    \item A cross-scan knowledge persistence system enabling progressive intelligence
    \item Comprehensive testing (808+ tests, 0 failures) and a fully resolved 83-finding security audit
\end{itemize}

\textbf{Keywords:} penetration testing, autonomous agents, large language models, cybersecurity, vulnerability assessment, Docker sandboxing, ReAct reasoning

\chapter{Introduction}
\label{ch:introduction}

\section{Motivation and Context}

The cybersecurity landscape has undergone a fundamental transformation. Attack surfaces have expanded dramatically with cloud-native architectures, microservices, APIs, and single-page applications. Meanwhile, the global shortage of skilled penetration testers---estimated at 3.4 million professionals by 2025 (ISC$^2$ Cybersecurity Workforce Study)---leaves organizations unable to assess their security posture with adequate frequency or depth.

Traditional vulnerability scanners, while useful for baseline checks, suffer from well-documented limitations:
\begin{itemize}
    \item \textbf{Signature dependence:} They can only detect vulnerabilities matching predefined patterns, missing novel or chained attack vectors.
    \item \textbf{High false positive rates:} Studies consistently show 40--70\% false positive rates, consuming valuable analyst time.
    \item \textbf{Shallow coverage:} Single-pass scanning cannot discover multi-step exploits that require sequential reasoning.
    \item \textbf{No contextual reasoning:} They lack the ability to interpret application behavior and adapt strategies accordingly.
\end{itemize}

These limitations create a critical gap: organizations need human-level penetration testing, but cannot scale human expertise to match the speed and breadth of modern software deployment.

\section{Project Objectives}

\phantom@{} was designed to bridge this gap by creating an autonomous AI agent that conducts penetration tests with the reasoning capabilities of a skilled human tester. The specific objectives are:

\begin{enumerate}
    \item \textbf{Autonomous operation:} Conduct end-to-end security assessments---from reconnaissance through exploitation to reporting---without human intervention.
    \item \textbf{Adaptive reasoning:} Use LLM-powered reasoning to analyze targets, plan attack strategies, and adapt based on observed results.
    \item \textbf{Verified findings:} Produce working proof-of-concept exploits for every confirmed vulnerability, eliminating false positives.
    \item \textbf{Security-first design:} Ensure the platform itself is secure through defense-in-depth, sandboxing, and strict scope enforcement.
    \item \textbf{Actionable reporting:} Generate MITRE ATT\&CK-enriched, compliance-mapped reports with concrete remediation guidance.
\end{enumerate}

\section{Layer Responsibilities}

\subsection{Interface Layer}

The interface layer provides three interaction modes:

\begin{itemize}
    \item \textbf{CLI (Command Line Interface):} Standard command-line invocation with flags for target, scan mode, profile, and instructions.
    \item \textbf{TUI (Terminal User Interface):} Rich interactive mode built with Textual, providing real-time scan visualization, agent tree, and live findings.
    \item \textbf{Streaming Parser:} Processes LLM responses token-by-token, extracting tool calls from partial XML streams without waiting for complete responses.
\end{itemize}

\subsection{Orchestration Layer}

Manages the lifecycle of a scan:

\begin{itemize}
    \item \textbf{Scan Profiles:} Predefined configurations (quick, standard, deep, stealth, api\_only) controlling iteration limits, tool selection, and timeout thresholds.
    \item \textbf{Scope Validator:} Ensures all tool executions target only authorized hosts and ports.
    \item \textbf{Cost Controller:} Tracks per-request and cumulative LLM costs with configurable budget ceilings.
    \item \textbf{Audit Logger:} HMAC-signed, append-only event log providing tamper-evident forensic trail.
\end{itemize}

\subsection{Agent Core}

The cognitive engine of \phantom@{} (detailed in Chapter~\ref{ch:agent}):

\begin{itemize}
    \item \textbf{BaseAgent (ReAct Loop):} The primary reasoning cycle: Observe $\rightarrow$ Reason $\rightarrow$ Plan $\rightarrow$ Act $\rightarrow$ Record.
    \item \textbf{State Machine:} Bounded agent state with memory limits, preventing context overflow.
    \item \textbf{LLM Client:} Abstraction over LiteLLM supporting 100+ providers with automatic retry, fallback, and key rotation.
    \item \textbf{Memory Compressor:} Progressively summarizes older conversation turns to stay within context limits.
    \item \textbf{Skills Engine:} Injects domain-specific knowledge (OWASP Top 10, common CVEs, tool usage guides) into agent prompts.
\end{itemize}

\subsection{Security Layer}

A defense-in-depth barrier between reasoning and execution (detailed in Chapter~\ref{ch:security}):

\begin{itemize}
    \item \textbf{Tool Firewall:} Validates every tool call against allowlists, argument constraints, and scope rules before execution.
    \item \textbf{Verification Engine:} Cross-validates findings through independent re-exploitation.
    \item \textbf{Scope Enforcement:} Runtime checks ensuring tools cannot reach unauthorized targets.
\end{itemize}

\subsection{Execution Layer}

All tool execution occurs inside Docker containers (detailed in Chapter~\ref{ch:runtime}):

\begin{itemize}
    \item \textbf{Docker Sandbox:} Ephemeral Kali-based container (14\,GB) with restricted capabilities.
    \item \textbf{Tool Server:} HTTP API running inside the container, authenticated with per-scan tokens.
    \item \textbf{30+ Security Tools:} nmap, nuclei, sqlmap, ffuf, httpx, katana, semgrep, nikto, gobuster, arjun, and more.
    \item \textbf{Playwright Browser:} Headless Chromium for JavaScript-heavy application testing.
\end{itemize}

\subsection{Output Layer}

Post-scan enrichment and reporting (detailed in Chapter~\ref{ch:reports}):

\begin{itemize}
    \item \textbf{Reports:} Structured JSON, styled HTML, and Markdown output.
    \item \textbf{Attack Graph:} NetworkX-based directed graph modeling attack paths.
    \item \textbf{MITRE ATT\&CK:} Automatic CWE, CAPEC, and technique-level enrichment.
    \item \textbf{Compliance Mapper:} OWASP Top 10, PCI DSS v4, and NIST 800-53 control mapping.
    \item \textbf{Nuclei Templates:} Auto-generated YAML templates for reproducible scanning.
\end{itemize}

\section{The ReAct Pattern}

\phantom@{} uses the \textbf{ReAct} (Reasoning + Acting) paradigm, where the agent alternates between thinking about observations and taking actions. This contrasts with chain-of-thought prompting (reasoning only) or simple tool-calling (acting without explicit reasoning).

Each iteration of the loop:
\begin{enumerate}
    \item \textbf{Observe:} The agent receives tool output from the previous action (or the initial scan context).
    \item \textbf{Reason:} The LLM analyzes the current state, identifies patterns, and determines what information is still needed.
    \item \textbf{Plan:} The agent selects a specific tool and constructs its arguments.
    \item \textbf{Act:} The tool call is validated by the firewall and executed in the sandbox.
    \item \textbf{Record:} Results are stored in the agent state and findings ledger.
    \item \textbf{Check Stop:} If the iteration budget is exhausted, time limit reached, or findings are complete, the agent finalizes and generates reports.
\end{enumerate}

\section{Memory Compression}

As the conversation history grows, \phantom@{} applies progressive \textbf{memory compression}:

\begin{enumerate}
    \item \textbf{Threshold Detection:} When the conversation exceeds the memory threshold (configurable per profile), compression is triggered.
    \item \textbf{Summarization:} Older turns are condensed into a brief summary preserving key findings, discovered endpoints, and tool results.
    \item \textbf{Retention:} The most recent $k$ turns are kept verbatim to maintain immediate context.
    \item \textbf{Cost Savings:} Compression reduces token usage by 40--60\% on long scans.
\end{enumerate}

\subsection{Layer 1: Scope Validator}

Before any tool execution, the Scope Validator ensures the target is within the authorized scope:
\begin{itemize}
    \item Parses the scan target into hostname, IP, port, and path components.
    \item All tool calls are checked against the scope allowlist.
    \item SSRF protection: blocks tool calls to internal networks, cloud metadata, and link-local addresses.
    \item Configurable explicit deny lists for sensitive ranges.
\end{itemize}

\subsection{Layer 2: Tool Firewall}

The Tool Firewall inspects every tool invocation before it reaches the sandbox:
\begin{itemize}
    \item \textbf{Allowlist:} Only pre-registered tools can be invoked.
    \item \textbf{Argument Validation:} Parameters checked against expected types, lengths, and patterns.
    \item \textbf{Command Injection Prevention:} Shell metacharacters stripped from tool arguments.
    \item \textbf{Rate Limiting:} Prevents excessive calls to any single tool.
\end{itemize}

\subsection{Layer 3: Docker Sandbox}

All offensive operations execute inside ephemeral Docker containers:
\begin{itemize}
    \item \textbf{No host filesystem access:} Container has no volume mounts to the host.
    \item \textbf{Restricted capabilities:} Only \texttt{NET\_ADMIN} and \texttt{NET\_RAW} (needed for nmap).
    \item \textbf{Automatic cleanup:} Container destroyed after scan completion.
    \item \textbf{Token-authenticated API:} Tool server requires a per-scan secret token.
    \item \textbf{Network isolation:} Container can only reach the target and host gateway.
\end{itemize}

\subsection{Layer 4: Cost Controller}

Prevents runaway spending:
\begin{itemize}
    \item Per-request cost ceiling (default: \$5.00 per LLM call).
    \item Global scan budget (default: \$25.00).
    \item Graceful degradation: agent completes current phase and generates partial report when budget exceeded.
\end{itemize}

\subsection{Layer 5: Time Limits}

\begin{itemize}
    \item \textbf{Per-tool timeout:} Configurable (default: 600s per sandbox execution).
    \item \textbf{Global scan timeout:} Enforced at the orchestration layer.
    \item \textbf{LLM timeout:} Individual request timeouts with retry logic.
\end{itemize}

\subsection{Layer 6: HMAC Audit Trail}

\begin{itemize}
    \item Every event (tool call, finding, agent creation) written to an append-only JSONL file.
    \item Each entry includes an HMAC signature computed with a per-scan key.
    \item Chain integrity: each entry references the hash of the previous entry.
    \item Enables post-hoc forensic analysis and tamper detection.
\end{itemize}

\subsection{Layer 7: Output Sanitizer}

\begin{itemize}
    \item Strips PII patterns (emails, SSNs, credit card numbers) from reports.
    \item Redacts API keys, tokens, and credentials discovered during scanning.
    \item Ensures report content is safe for distribution.
\end{itemize}

\section{Attack Graph}

The attack graph is a directed acyclic graph (DAG) built with NetworkX:

\begin{itemize}
    \item \textbf{Nodes:} Hosts, services, endpoints, and vulnerabilities.
    \item \textbf{Edges:} Attack steps connecting nodes (e.g., host $\rightarrow$ service $\rightarrow$ endpoint $\rightarrow$ vulnerability).
    \item \textbf{Risk Scoring:} Each path scored by composite CVSS, exploitability, and impact.
    \item \textbf{Prioritization:} Critical paths surfaced to remediation recommendations.
\end{itemize}

\section{Scan Resume (Checkpoint System)}

\phantom@{} supports scan resume through a checkpoint system:

\begin{infobox}[Checkpoint System]
\begin{enumerate}
    \item \textbf{Periodic Saving:} The \texttt{EnhancedAgentState} is serialized to \texttt{checkpoint.json} every 10 iterations.
    \item \textbf{State Captured:} Iteration count, phase, discovered hosts, endpoints, tested URLs, vulnerability IDs, and tool usage.
    \item \textbf{Resume Process:} On restart with \texttt{--resume}, the checkpoint is loaded and the agent continues from the saved iteration with remaining budget.
    \item \textbf{Sandbox Recreation:} A new container is created, but the agent state carries forward.
\end{enumerate}
\end{infobox}

\chapter{Future Work}
\label{ch:future}

\section{Short-Term Roadmap (v1.0)}

\begin{enumerate}
    \item \textbf{Multi-Target Scanning:} Parallel scanning of multiple hosts, domains, or IP ranges in a single session.
    \item \textbf{SARIF Export:} Standard SARIF format for native GitHub Code Scanning integration.
    \item \textbf{Plugin System:} Community-contributed tool integrations via a plugin API.
    \item \textbf{Dashboard UI:} Web-based dashboard for scan management, result visualization, and team collaboration.
    \item \textbf{Scan Scheduling:} Cron-based recurring scans with delta reporting.
\end{enumerate}

\section{Medium-Term Research}

\begin{enumerate}
    \item \textbf{Reinforcement Learning from Scan Feedback:} Train specialized models on successful/failed attack strategies to improve tool selection.
    \item \textbf{Collaborative Multi-Agent Reasoning:} Advanced agent coordination with shared working memory and inter-agent communication.
    \item \textbf{Adaptive Scan Profiles:} AI-selected scan profiles based on target fingerprinting.
    \item \textbf{Vulnerability Chaining:} Automatic multi-step exploit chain construction (e.g., SSRF $\rightarrow$ internal service $\rightarrow$ RCE).
    \item \textbf{Custom LLM Fine-Tuning:} Domain-specific fine-tuning on penetration testing datasets.
\end{enumerate}

\section{Long-Term Vision}

\begin{enumerate}
    \item \textbf{Continuous Security Agent:} Always-on agent monitoring applications for regressions and new vulnerabilities.
    \item \textbf{Adversary Simulation:} Full red-team operations covering network, application, and social engineering vectors.
    \item \textbf{Autonomous Remediation:} Agent-generated pull requests fixing discovered vulnerabilities.
    \item \textbf{Threat Intelligence Integration:} Real-time integration with CVE feeds, exploit databases, and threat intelligence platforms.
\end{enumerate}
