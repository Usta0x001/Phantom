% -- Abstract ----------------------------------------------
\begin{abstract}
\noindent
Phantom is an autonomous AI-powered penetration testing agent that performs full web application security assessments without human intervention. Given a target URL and an LLM API key, Phantom spawns a tree of specialized agents, each capable of reasoning about the target, selecting and executing security tools, interpreting results, and escalating findings. All tool execution occurs inside an isolated Docker sandbox containing more than 20 professional security tools. Phantom produces structured reports in HTML, Markdown, JSON, SARIF, and PDF formats, with full MITRE ATT\&CK and OWASP Top 10 mapping. The current release (v0.8.1) ships with 146 passing tests, is published on PyPI, and supports any LLM provider via LiteLLM.
\end{abstract}

\section{System Overview}
% ============================================================

Phantom is a complete, self-contained penetration testing platform consisting of two parts:

\begin{enumerate}
  \item \textbf{Phantom CLI} \textemdash{} the Python package installed by the user (\texttt{pip install phantom-agent}). Handles configuration, LLM communication, agent orchestration, and reporting.
  \item \textbf{Phantom Sandbox Image} \textemdash{} a Docker image (\texttt{ghcr.io/usta0x001/phantom-sandbox:latest}) containing the tool server and all security tools. Pulled and launched automatically per scan.
\end{enumerate}

The two components communicate over an authenticated HTTP channel. The user never manages the sandbox directly.

\subsection{Key Design Principles}

\begin{description}
  \item[Zero human-in-the-loop] Once launched, Phantom runs to completion without asking for input.
  \item[Proof-of-concept first] Every reported vulnerability is accompanied by a working exploit demonstration.
  \item[Portable by design] The CLI installs via \texttt{pip}; the sandbox image is pulled automatically. No manual setup required.
  \item[Provider-agnostic] Any LLM supported by LiteLLM (OpenAI, Anthropic, Google, Groq, Ollama, \ldots) works out of the box.
  \item[Sandboxed execution] All tool invocations run inside an isolated Docker container. The host system is never touched.
\end{description}

\subsection{Agent Lifecycle}

Each agent follows this deterministic state machine:

\begin{enumerate}
  \item \textbf{INIT} --- Receive system prompt + current KnowledgeGraph context.
  \item \textbf{PLAN} --- LLM call: generate a step-by-step action plan.
  \item \textbf{EXECUTE} --- For each step: call tool via HTTP to sandbox, parse output, update KG.
  \item \textbf{REFLECT} --- LLM call: evaluate results, determine if sub-agents are needed.
  \item \textbf{SPAWN} --- Optionally spawn child agents with sub-tasks.
  \item \textbf{REPORT} --- Return structured findings to parent.
  \item \textbf{DONE} --- Agent terminates, resources released.
\end{enumerate}

\subsection{Context \& Memory Management}

\begin{itemize}
  \item \textbf{In-context memory}: the current LLM context window, managed by a compression module that summarizes older turns when approaching token limits.
  \item \textbf{Persistent memory}: the KnowledgeGraph JSON, shared across all agents in a scan session.
\end{itemize}

\subsection{Security Isolation}

\begin{itemize}
  \item Tool execution runs as the \texttt{pentester} user inside the container (not root).
  \item The container has no persistent volumes; all data is ephemeral.
  \item A unique per-session bearer token is generated and injected; the server rejects unauthenticated requests (HTTP 401).
  \item The container runs on the default bridge network: reachable to the target, isolated from host internal services.
\end{itemize}

\subsection{Step-by-Step Execution Trace}

\subsubsection{Step 1 --- CLI Invocation (L0)}

\begin{lstlisting}[language=bash]
phantom scan --target https://app.example.com \
             --scan-mode quick --non-interactive
\end{lstlisting}

Typer parses arguments, validates the target URL is reachable, displays an authorization confirmation, and in interactive mode launches the Textual TUI.

\subsubsection{Step 2 --- Configuration Resolution (L1)}

Settings are resolved in priority order:
\begin{enumerate}
  \item Environment variables: \texttt{PHANTOM\_LLM}, \texttt{LLM\_API\_KEY}, \texttt{PHANTOM\_IMAGE}
  \item Persistent config: \texttt{\textasciitilde{}/.phantom/config.json}
  \item Built-in defaults: \texttt{scan\_mode=standard}, \texttt{max\_iterations=300}
\end{enumerate}
Pydantic v2 validates the final config; raises a descriptive error if the LLM model is not set.

\subsubsection{Step 3 --- Sandbox Startup (L2)}

\begin{lstlisting}[language=Python, caption={Container lifecycle inside docker\_runtime.py}]
# 1. Check if the image exists locally; if not, docker pull
# 2. Generate a random per-session token (secrets.token_urlsafe(32))
# 3. docker run -d
#        -e TOOL_SERVER_TOKEN=<token>
#        -e TOOL_SERVER_PORT=48081
#        -p <random_host_port>:48081
#        ghcr.io/usta0x001/phantom-sandbox:latest
# 4. Poll GET /health every 2s (timeout: 60s)
# 5. Return SandboxInfo(container_id, host_port, token)
\end{lstlisting}

The token is never written to disk. It is injected as an environment variable and purged with the container.

\subsubsection{Step 4 --- Root Agent Initialization (L3)}

\begin{lstlisting}[language=Python]
state = EnhancedAgentState(
    agent_name    = "Phantom Root Agent",
    task          = f"Pentest: {target}",
    sandbox_id    = sandbox.container_id,
    sandbox_token = sandbox.token,
)
orchestrator = PhantomAgent(config=config, state=state)
orchestrator.run()   # Begins the agentic loop
\end{lstlisting}

\subsubsection{Step 5 --- Reconnaissance Phase (L3 + L4 + L5 + L6)}

\begin{enumerate}
  \item \textbf{LLM call (L4):} Root asks the LLM what to do first; LLM returns \texttt{spawn recon\_agent(target)}.
  \item \textbf{Recon Agent} activates and begins its own planning loop.
  \item \textbf{Tool calls (L5/L6):}
\begin{lstlisting}[language=bash, numbers=none]
POST /execute  {"tool_name":"terminal_execute",
                "kwargs":{"command":"nmap -sV -p 80,443,8080 target"}}
POST /execute  {"tool_name":"terminal_execute",
                "kwargs":{"command":"whatweb --color=never target"}}
POST /execute  {"tool_name":"terminal_execute",
                "kwargs":{"command":"ffuf -u target/FUZZ -w /opt/seclists/common.txt"}}
\end{lstlisting}
  \item \textbf{KG update (L7):} Discovered ports, technologies, endpoints written to graph.
  \item Recon Agent returns structured summary to Root Agent.
\end{enumerate}

\subsubsection{Step 6 --- Vulnerability Analysis Phase (L3 + L4)}

The \textbf{Vulnerability Analysis Agent} receives full KG context and reasons about likely vulnerability classes given the tech stack (e.g., WordPress $\rightarrow$ plugin CVEs; PHP + direct SQL $\rightarrow$ injection risk). No tools are invoked --- this is pure LLM reasoning producing a prioritised \texttt{VulnerabilityCandidate} list.

\subsubsection{Step 7 --- Exploitation Phase (L3 + L4 + L5 + L6)}

For each candidate, an \textbf{Exploitation Agent} is spawned:

\begin{lstlisting}[language=bash, numbers=none, caption={Example exploitation calls}]
# SQL injection
POST /execute {"kwargs":{"command":"sqlmap -u 'target/login?id=1' --dbs --batch"}}
# CVE template scan
POST /execute {"kwargs":{"command":"nuclei -u target -t cves/ -severity critical,high"}}
# XSS probe
POST /execute {"kwargs":{"command":"curl -s 'target/search?q=<script>alert(1)</script>'"}}
\end{lstlisting}

Confirmed findings become \texttt{Vulnerability} objects with CVSS score, evidence, and PoC. In deep mode, a \textbf{Post-Exploitation Agent} may be spawned to measure impact.

\subsubsection{Step 8 --- Validation Phase (L3 + L4)}

A \textbf{Validation Agent} re-executes the minimal reproducer for each finding. False positives are marked and excluded from the final report. Confirmed findings move to \texttt{KnowledgeGraph.verified\_vulns}.

\subsubsection{Step 9 --- Report Generation (L8)}

\begin{lstlisting}[language=Python]
engine = ReportEngine(knowledge_graph)
engine.render_html()     # phantom_runs/<id>/report.html
engine.render_markdown() # phantom_runs/<id>/report.md
engine.render_json()     # phantom_runs/<id>/report.json
engine.render_sarif()    # phantom_runs/<id>/report.sarif
engine.render_pdf()      # phantom_runs/<id>/report.pdf
\end{lstlisting}

\subsubsection{Step 10 --- Sandbox Teardown (L2)}

\begin{lstlisting}[language=bash, numbers=none]
docker stop  phantom-sandbox-<uuid>
docker rm    phantom-sandbox-<uuid>
# Token purged; all ephemeral container data destroyed
\end{lstlisting}

\section{Knowledge Representation}
% ============================================================

\subsection{Knowledge Graph}

All information discovered during a scan is stored in a shared \textbf{KnowledgeGraph} updated in real time by all active agents:

\begin{itemize}
  \item \textbf{Hosts \& Services} --- IPs, open ports, service banners, TLS certificates.
  \item \textbf{Endpoints} --- URLs, HTTP methods, parameters, authentication requirements.
  \item \textbf{Technologies} --- Frameworks, CMS, server software, version numbers.
  \item \textbf{Vulnerabilities} --- \texttt{Vulnerability} objects with CVE, CVSS, OWASP, MITRE, evidence, PoC.
  \item \textbf{Attack chains} --- Directed edges between chainable vulnerabilities.
\end{itemize}

\subsection{Vulnerability Data Model}

\begin{lstlisting}[language=Python, caption={Simplified Vulnerability Pydantic model}]
class Vulnerability(BaseModel):
    id:               str            # SHA-256 fingerprint
    title:            str
    severity:         Severity       # CRITICAL/HIGH/MEDIUM/LOW/INFO
    cvss_score:       float          # 0.0 - 10.0
    cvss_vector:      str            # CVSS v3.1 vector string
    cve_id:           Optional[str]  # CVE-YYYY-NNNN
    owasp:            str            # e.g. "A03:2021 - Injection"
    mitre_ttp:        str            # e.g. "T1190"
    endpoint:         str
    parameter:        Optional[str]
    evidence:         str            # Raw tool output
    proof_of_concept: str            # Reproducer command/payload
    remediation:      str
    references:       List[str]
    first_seen:       datetime
    last_seen:        datetime
\end{lstlisting}

\section{Security \& Ethical Considerations}
% ============================================================

\subsection{Authorization Enforcement}

Before any scan, Phantom: (1) displays a legal warning, (2) requires explicit confirmation, (3) validates target URL and scope.

\subsection{Scope Validation}

Phantom enforces strict scope. Any tool call targeting an out-of-scope address is blocked before execution.

\subsection{Data Privacy}

Scan data is stored locally only (\texttt{phantom\_runs/}). The LLM receives tool outputs and target responses; the tool server token is never included in prompts.

\subsection{Responsible Use}

Phantom is intended for: authorised internal security assessments, bug bounty hunters on in-scope targets, developers testing their own applications, and academic research on dedicated lab environments.

\begin{warnbox}
Use against systems without written authorisation is illegal under computer fraud and abuse laws in most jurisdictions.
\end{warnbox}

\section{Conclusion}
% ============================================================

Phantom is a complete, production-ready autonomous penetration testing agent with the following key properties:

\begin{itemize}
  \item \textbf{Full autonomy}: target URL to structured report with zero human intervention.
  \item \textbf{Real exploitation}: findings are backed by working PoC code, not theoretical assessments.
  \item \textbf{Clean two-component architecture}: CLI on host; all offensive tools inside an isolated Docker sandbox.
  \item \textbf{Provider flexibility}: LiteLLM abstraction --- any LLM provider substitutable without code changes.

Future directions: multi-target parallel scanning, authenticated application testing (cookies/CSRF), mobile and REST API scan modes, web dashboard for distributed scan management.
