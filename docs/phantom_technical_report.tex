% -- Abstract ----------------------------------------------
\begin{abstract}
\noindent
Phantom is an autonomous AI-powered penetration testing agent that performs full web application security assessments without human intervention. Given a target URL and an LLM API key, Phantom spawns a tree of specialized agents, each capable of reasoning about the target, selecting and executing security tools, interpreting results, and escalating findings. All tool execution occurs inside an isolated Docker sandbox containing more than 20 professional security tools. Phantom produces structured reports in HTML, Markdown, JSON, SARIF, and PDF formats, with full MITRE ATT\&CK and OWASP Top 10 mapping. The current release (v0.8.1) ships with 146 passing tests, is published on PyPI, and supports any LLM provider via LiteLLM.
\end{abstract}

\section{System Overview}
% ══════════════════════════════════════════════════════════

Phantom is a complete, self-contained penetration testing platform. It consists of two parts:

\begin{enumerate}
  \item \textbf{Phantom CLI} \textemdash{} the Python package installed by the user (\texttt{pip install phantom-agent}). Handles configuration, LLM communication, agent orchestration, and reporting.
  \item \textbf{Phantom Sandbox Image} \textemdash{} a Docker image (\texttt{ghcr.io/usta0x001/phantom-sandbox:latest}) containing the tool server and all security tools (nmap, sqlmap, nuclei, ffuf, etc.). Launched automatically per scan.
\end{enumerate}

The two components communicate over an encrypted HTTP channel. The user never touches the sandbox directly.

\subsection{Key Design Principles}

\begin{description}
  \item[Zero human-in-the-loop] Once launched, Phantom runs to completion without asking for input.
  \item[Proof-of-concept first] Every reported vulnerability is accompanied by a working exploit demonstration.
  \item[Portable by design] The CLI installs via \texttt{pip}; the sandbox image is pulled automatically. No manual setup.
  \item[Provider-agnostic] Any LLM supported by LiteLLM (OpenAI, Anthropic, Google, Groq, Ollama, \ldots) works out of the box.
  \item[Sandboxed execution] All tool invocations run inside an isolated Docker container. The host system is never touched.
\end{description}

\subsection{Agent Lifecycle}

Each agent follows this deterministic state machine:

\begin{enumerate}
  \item \textbf{INIT} \textemdash{} Receive system prompt + current KnowledgeGraph context.
  \item \textbf{PLAN} \textemdash{} LLM call: generate a step-by-step action plan.
  \item \textbf{EXECUTE} \textemdash{} For each step: call tool via HTTP to sandbox, parse output, update KG.
  \item \textbf{REFLECT} \textemdash{} LLM call: evaluate results, determine if sub-agents are needed.
  \item \textbf{SPAWN} \textemdash{} Optionally spawn child agents with sub-tasks.
  \item \textbf{REPORT} \textemdash{} Return structured findings to parent.
  \item \textbf{DONE} \textemdash{} Agent terminates, resources released.
\end{enumerate}

\subsection{Context \& Memory Management}

Agents have two types of memory:

\begin{itemize}
  \item \textbf{In-context memory}: the current LLM context window, managed by a compression module that summarises older turns when approaching token limits.
  \item \textbf{Persistent memory}: the KnowledgeGraph JSON, which survives agent restarts and is shared across all agents in a scan session.
\end{itemize}

The memory compressor uses an LLM call to condense historical tool outputs into concise summaries without losing critical vulnerability data.

The tool server exposes a simple HTTP API used by all agents:

\begin{lstlisting}[language=bash, caption={Tool Server API endpoints}]

-> {"status": "healthy", "sandbox_mode": true, "active_agents": 0}

# Execute a tool
POST /execute
Authorization: Bearer <TOOL_SERVER_TOKEN>
Content-Type: application/json

{
  "agent_id": "recon-001",
  "tool_name": "terminal_execute",
  "kwargs": {
    "command": "nmap -sV -p 80,443,8080 target.example.com",
    "timeout": 120
  }
}

-> {
  "content": "[recon-001]$ nmap ...\nNmap scan report ...\n",
  "exit_code": 0
}
\end{lstlisting}

\subsection{Security Isolation}

\begin{itemize}
  \item Tool execution is performed by the \texttt{pentester} user inside the container (not root).
  \item The container has no persistent volumes; all data is ephemeral unless explicitly exported.
  \item A unique per-session bearer token is generated and injected; the tool server rejects all unauthenticated requests.
  \item The container runs with \texttt{--net=bridge} (default) and can reach the target but not the host's internal services unless explicitly configured.
\end{itemize}

\subsection{Provider-Agnostic Design}

\subsection{Retry \& Fallback Chain}

Phantom automatically retries failed LLM calls with exponential backoff (using the \texttt{tenacity} library). If a primary model is unavailable, a configured fallback chain is tried in order. This ensures scan continuity in case of rate limits or API errors.

\subsection{Differential Scanning}

When run against the same target multiple times, Phantom compares findings to the previous run and produces a \emph{diff report} showing:
\begin{itemize}
  \item \textbf{New vulnerabilities} found since the last scan;
  \item \textbf{Fixed vulnerabilities} that no longer reproduce;
  \item \textbf{Persistent vulnerabilities} still present.
\end{itemize}

This is particularly useful for development pipelines and re-test verification.

\section{Knowledge Representation}
% ══════════════════════════════════════════════════════════

\subsection{Knowledge Graph}

All information discovered during a scan is stored in an in-memory \textbf{KnowledgeGraph} that agents read from and write to in real time.

The graph captures:
\begin{itemize}
  \item \textbf{Hosts \& Services} \textemdash{} IP addresses, open ports, service banners, TLS certificates.
  \item \textbf{Endpoints} \textemdash{} URLs, HTTP methods, parameters, headers, authentication requirements.
  \item \textbf{Technologies} \textemdash{} Frameworks, CMS, server software, version numbers.
  \item \textbf{Vulnerabilities} \textemdash{} Each as a \texttt{Vulnerability} Pydantic object with CVE ID (if applicable), CVSS score, OWASP category, MITRE TTP, evidence, PoC, remediation.
  \item \textbf{Attack chains} \textemdash{} Directed edges between vulnerabilities that can be chained.
\end{itemize}

\subsection{Vulnerability Data Model}

\begin{lstlisting}[language=Python, caption={Simplified Vulnerability Pydantic model}]
class Vulnerability(BaseModel):
    id:          str            # SHA-256 fingerprint
    title:       str            # Human-readable name
    severity:    Severity       # CRITICAL / HIGH / MEDIUM / LOW / INFO
    cvss_score:  float          # 0.0 -- 10.0
    cvss_vector: str            # CVSS v3.1 vector string
    cve_id:      Optional[str]  # CVE-YYYY-NNNN
    owasp:       str            # e.g. "A03:2021 - Injection"
    mitre_ttp:   str            # e.g. "T1190 - Exploit Public-Facing App"
    endpoint:    str            # Affected URL / service
    parameter:   Optional[str]  # Affected parameter
    evidence:    str            # Raw tool output proving the finding
    proof_of_concept: str       # Reproducer / curl command / payload
    remediation: str            # Fix recommendation
    references:  List[str]      # CVE, OWASP, NVD links
    first_seen:  datetime
    last_seen:   datetime
\end{lstlisting}

\section{Security \& Ethical Considerations}
% ══════════════════════════════════════════════════════════

\subsection{Authorization Enforcement}

Before any scan begins, Phantom:
\begin{enumerate}
  \item Displays a clear warning that scanning without authorization is illegal;
  \item Requires the user to explicitly confirm authorization (interactive prompt or \texttt{--instruction} parameter);
  \item Validates the target URL format and scope boundaries.
\end{enumerate}

\subsection{Scope Validation}

Phantom enforces a strict scope system. Agents are only permitted to interact with:
\begin{itemize}
  \item The explicitly specified target URL and its subdomains/paths;
  \item IPs within the declared CIDR range (if provided).
\end{itemize}
Any tool call targeting an out-of-scope address is blocked before execution.

\subsection{Data Privacy}

\begin{itemize}
  \item No scan data is transmitted to third parties (other than the LLM API for agent reasoning).
  \item All results are stored locally in the \texttt{phantom\_runs/} directory.
  \item The LLM receives tool outputs and target responses; the tool server token is never exposed to the LLM.
\end{itemize}

\subsection{Responsible Disclosure}

Phantom is intended for:
\begin{itemize}
  \item Security teams performing authorised internal assessments;
  \item Bug bounty hunters on in-scope targets;
  \item Developers testing their own applications;
  \item Academic security research.
\end{itemize}

Use against systems without written permission is illegal under computer fraud laws in most jurisdictions.

\section{Conclusion}
% ══════════════════════════════════════════════════════════

Phantom is a complete, production-ready autonomous penetration testing agent. Its key properties are:

\begin{itemize}
  \item \textbf{Full autonomy}: from target URL to structured report with zero human intervention.
  \item \textbf{Real exploitation}: findings are backed by working proof-of-concept code, not theoretical assessments.
  \item \textbf{Clean architecture}: a two-component design (CLI + sandbox) keeps the host clean while providing a full Kali Linux tooling environment.
  \item \textbf{Provider flexibility}: LiteLLM abstraction means any LLM backend can be substituted without code changes.
  \item \textbf{Proven correctness}: 146/146 unit tests passing; sandbox tool execution verified end-to-end.
  \item \textbf{Easy distribution}: \texttt{pipx install phantom-agent} is the single install step; the sandbox image is auto-pulled on demand.
\end{itemize}

Future directions include: multi-target parallel scanning, authenticated application testing, mobile/API-first scan modes, and a web dashboard for managing scan sessions.
